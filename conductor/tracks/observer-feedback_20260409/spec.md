## Summary

Webhook do GitHub notifica o Observer quando um PR e mergeado ou fechado. O Observer atualiza a tabela de diagnosticos com o resultado (resolveu/nao resolveu), criando um loop de feedback que melhora a confianca do sistema ao longo do tempo.

## Context

Hoje o Observer cria PRs mas nunca sabe se o fix realmente resolveu o problema. Sem feedback, nao ha como medir eficacia, ajustar confianca, ou aprender com erros. O feedback loop fecha esse ciclo.

## User Story

Como gestor do pipeline, quero saber a taxa de sucesso dos PRs do Observer (quantos foram mergeados vs fechados), para avaliar a eficacia do agente e ajustar parametros.

## Acceptance Criteria

- [ ] Webhook do GitHub configurado para eventos `pull_request` (merged, closed)
- [ ] Tabela `observer.diagnostics` atualizada com resultado do PR
- [ ] Metricas de sucesso: taxa de merge, tempo medio ate merge, taxa de rejeicao
- [ ] Dashboard mostra eficacia do Observer ao longo do tempo

## Dependencies

## Out of Scope

- Ajuste automatico de parametros baseado em feedback (futuro)
- Re-diagnostico automatico quando PR e rejeitado

## Technical Notes

### Opcao A: GitHub Webhook + Lambda

GitHub envia webhook para Lambda/Cloud Function que atualiza a tabela Delta.
Requer infraestrutura extra (API Gateway + Lambda).

### Opcao B: Polling Periodico

Job Databricks que roda periodicamente e verifica status dos PRs abertos via GitHub API.
Mais simples, sem infra extra, mas com latencia.

### Opcao C: GitHub Actions

GitHub Action que roda no evento `pull_request` e chama Databricks API para atualizar tabela.
Sem infra extra, usa a infra de CI/CD existente.

### Recomendacao

Comecar com Opcao C (GitHub Actions) por simplicidade e zero infra extra.
