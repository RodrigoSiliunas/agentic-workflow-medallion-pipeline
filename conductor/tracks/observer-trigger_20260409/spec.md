# Specification: Observer — Trigger Automatico

**Track ID:** observer-trigger_20260409
**Type:** Feature
**Created:** 2026-04-09
**Status:** Draft
**Priority:** 1 (mais alta)

## Summary

Fazer o Observer Agent ser disparado automaticamente quando o pipeline falha, ao inves de depender de polling ou trigger manual. Usar Webhook Notifications do Databricks ou task final com `run_if: AT_LEAST_ONE_FAILED` que chama a Jobs API.

## Context

Hoje o Observer precisa ser disparado manualmente via SDK ou pelo `agent_post`. O ideal e que o Databricks notifique automaticamente quando um run falha, disparando o Observer com o `run_id` da falha. Isso reduz o tempo entre falha e diagnostico de minutos para segundos.

## User Story

Como engenheiro de dados, quero que o Observer seja disparado automaticamente quando qualquer workflow falhar, para que o diagnostico e correcao sejam iniciados imediatamente sem intervencao humana.

## Acceptance Criteria

- [ ] Observer e disparado automaticamente quando pipeline ETL falha
- [ ] O `run_id` da run que falhou e passado automaticamente ao Observer
- [ ] Latencia entre falha e inicio do Observer < 60 segundos
- [ ] Pipeline normal (sem falha) NAO dispara o Observer
- [ ] Funciona com qualquer workflow do workspace (nao so o ETL)

## Dependencies

Nenhuma — pode ser implementada independentemente.

## Technical Notes

### Opcao A: Task Sentinel no Workflow

Adicionar task final `observer_trigger` no workflow ETL com `run_if: AT_LEAST_ONE_FAILED`.
Essa task chama `w.jobs.run_now(job_id=OBSERVER_JOB_ID, notebook_params={"source_run_id": run_id})`.

Vantagens: simples, nativo Databricks, zero infraestrutura extra.
Desvantagens: acoplado ao workflow (precisa adicionar task em cada workflow).

### Opcao B: Webhook Notifications

Configurar webhook no workflow que notifica uma Cloud Function/Lambda quando run termina.
A funcao filtra por `result_state=FAILED` e dispara o Observer.

Vantagens: desacoplado, funciona com qualquer workflow sem modificar.
Desvantagens: infraestrutura extra (Lambda + API Gateway).

### Recomendacao

Comecar com Opcao A (task sentinel) por simplicidade. Migrar para Opcao B quando houver multiplos workflows.
